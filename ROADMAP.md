### 1.1 아키텍처 요약

- **순수 정적 SPA**: 빌드 도구·프레임워크·npm 런타임 의존성 **0**. 바닐라 ES 모듈 19개 + classic script 2개(data.js, boot.js) + CSS 1파일.
- **렌더링**: 해시 라우터(`#evalplan:grade` 식) → `render()`가 `#main.innerHTML` 전체 교체. 상태 = 모듈 스코프 변수 + localStorage.

- **보안**: CSP `script-src 'self'`(인라인 전면 금지), 이벤트는 전량 `data-on*` 위임(utils `registerActions`), Gemini 키는 헤더 전송.

## 7. 제품 로드맵 — 기능 개선·업데이트 (2026-09 ~ 2027-03)

### 7.0 경쟁 지형 변화 대응 (2026-07-09 확정 — 전략 축)

**계기**: 경기도교육청이 '평가계획 딸깍(자동 생성)' 도구를 출시. "성취기준 복붙"이 상품화(commodity)됨.

**해석 (좌절 금지 근거)**: ① 성취기준 복붙은 정글의 해자가 아니라 **현관문**이었다 — 원래부터 베끼기 쉬운 기능. 정글의 실가치는 그 뒤의 통합 워크플로(개요→담기→수업/평가→루브릭→정기시험→성적산출→차시)와 **정보 교과 특화·속도·취향**이다. ② 관(官) 도구 출시는 비석이 아니라 **시장 검증**(수요 실재 증명). 관 도구는 느리고·범용이고·못생기고·경기도 울타리에 묶임 — 개인 프로젝트의 속도/취향/니치를 못 따라온다. ③ 정글은 스타트업이 아니라 **개인 크래프트+포트폴리오**(정보카츄 포함) — 성공 척도가 "관보다 점유율 높은가"가 아니다.

**전략 전환 (자원 배분 원칙)**:
- **평가계획 '생성' 기능 패리티 경쟁 하지 말 것** — 예산 있는 쪽이 이기는 지는 게임. 관련 백로그(묶음 B의 평가계획 폼 등) 우선순위 하향.
- **관이 못 오는 축으로 이동**: ⓐ **정보 교과 특화**(성적산출 5등급·코드 변형·AI 아이디어 등 범용 도구가 최적화 안 하는 것) ⓑ **연동>경쟁**(관 도구가 평가계획을 뱉으면 정글은 다듬고 잇는 곳 — 평가계획→루브릭 시드, 정기시험 연동) ⓒ **개인 아카이브/포트폴리오**(정보카츄) ⓓ **속도·취향**(하루 한 파도).

- **만들 것**: "세특 프롬프트 빌더" — 담은 성취기준 + 평가 항목 + 관찰 키워드 몇 개를 입력받아 **훈령 준수 프롬프트**(금지 표현·글자수 규격 포함)를 조립해 클립보드 복사. 교사가 자기 ChatGPT/Claude/Gemini에 붙여넣음.
- **장점**: API 키 불필요(4원칙 2 통과), 학생 데이터가 사이트에 저장 안 됨(3 통과), 어떤 LLM과도 호환, 개발 비용 소.
- **만들지 말 것**: 사이트 내 직접 생성(API 종속·비용·개인정보 3중 위반).

### 8.3 GitHub 전면 업로드(git 도입)의 이점 — **✅ 도입 완료(2026-07-09)**, 아래 표는 도입 근거 기록

현 방식: 로컬 폴더가 단일 원본, GitHub 웹으로 수동 업로드. 문제는 §1.4-1(부분 업로드 사고 전력, git 부재).

| # | 이점 | 정글에 주는 실질 가치 |
|---|---|---|
| 1 | **부분 업로드 사고 원천 차단** | commit+push = 원자적 배포. 2026-07-07 유형 사고(파일 조합 불일치) 구조적으로 불가능해짐 — **도입 이유의 8할** |
| 2 | **원클릭 롤백** | 배포 후 문제 발견 시 `git revert` 한 번. 지금은 "이전 파일이 뭐였는지" 기억에 의존 |
| 3 | **로컬 소실 방어** | 지금은 이 PC 고장 = 최신본 소실(GitHub엔 구버전). push 습관이 곧 오프사이트 백업 |
| 4 | **변경 이력 자동화** | CLAUDE.md '완료됨' 수기 기록을 diff가 보완 — 후임 모델이 "언제 뭐가 바뀌었나"를 스스로 조회 |
| 5 | **verify 자동 게이트** | GitHub Actions(무료)로 push마다 `node tools/verify.mjs` 자동 실행 — 사람이 잊어도 기계가 잡음 |
| 6 | **후임 모델 작업 품질** | git이 있어야 Claude Code의 diff 기반 워크플로(/code-review, 변경 추적)가 작동 — 하위 모델 위임 안전성 직결 |
| 7 | 기여 통로 | 다른 정보교사의 오탈자·자료 PR 가능(선택적) |

**비용/리스크**: 0원(이미 GitHub 사용). 이력 공개는 신규 리스크 아님(폴더 전체가 이미 공개물, 코드에 비밀 없음 확인). 유일한 전제 = 이 PC에 git 설치(§1.4 환경 제약). **도입하는 날의 작업 순서**: git 설치 → 기존 원격 저장소 clone → 로컬 최신본 덮어쓰기 → 첫 커밋(“v19 baseline”) → 이후 배포는 push로 일원화 + §6 체크리스트의 업로드 항목을 push로 대체.
